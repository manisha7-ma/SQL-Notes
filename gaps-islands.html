01 · Foundation

What are Gaps & Islands?

Imagine a roll call. Students are numbered 1–20. Some are absent. The gaps are the absent students — missing numbers in a sequence. The islands are the groups of consecutive present students — like students 1,2,3 sitting together, then 7,8,9 as another cluster.

In SQL, Gaps & Islands refers to a family of problems involving sequences — usually of numbers or dates. You want to find:

Gaps — ranges that are missing from the data (the holes in the sequence).
Islands — consecutive ranges that exist in the data (the clusters/groups).

02 · Why It Matters

Why is this hard?

SQL is great at comparing rows to a fixed value. It's much trickier to compare a row to its neighbour — because tables are fundamentally unordered sets, not ordered lists. Gaps & Islands forces you to think about relative position, which is exactly what window functions were built for.

The naive approach — looping through rows in application code — is slow and defeats the purpose of having a database. The SQL approach uses window functions to solve this in a single query, elegantly and efficiently.

03 · The Core Technique

The id − row_number Magic

The most famous technique for finding islands uses a beautifully simple mathematical insight:

💡 The Key Insight

If a sequence of numbers is consecutive, the difference between the value and its row number is constant. When there's a gap, that constant changes — creating a new group.

Same difference = same island group. Different difference = different island. Now GROUP BY (id - row_number) gives you each island!

Step-by-Step Walkthrough

1
Start with your data

A table of IDs or dates with some values missing from the sequence.

2
Add ROW_NUMBER()

Order the rows by the sequence column and number them 1, 2, 3... using a window function.

3
Compute the difference

Subtract the row number from the sequence value. Consecutive rows produce the same difference.

4
GROUP BY the difference

Each unique difference value represents one island. Use MIN() and MAX() to get island boundaries.

Full Example — The Basic Islands Query

MySQL 8+ & SQL ServerSetup — sample data
-- Create a sample table with gaps
CREATE TABLE numbers (
  id INT PRIMARY KEY
);

INSERT INTO numbers VALUES
  (1),(2),(3),            -- Island 1
  (6),(7),              -- Island 2  (gap: 4,5 missing)
  (10),(11),(12);       -- Island 3  (gap: 8,9 missing)
MySQL 8+ & SQL ServerStep 1 — See the raw difference
SELECT
  id,
  ROW_NUMBER() OVER (ORDER BY id) AS rn,
  id - ROW_NUMBER() OVER (ORDER BY id) AS grp
FROM numbers;

-- Result:
-- id  | rn | grp
-- ----+----+----
-- 1   |  1 |  0   ← Island 1 group (grp=0)
-- 2   |  2 |  0
-- 3   |  3 |  0
-- 6   |  4 |  2   ← Island 2 group (grp=2)
-- 7   |  5 |  2
-- 10  |  6 |  4   ← Island 3 group (grp=4)
-- 11  |  7 |  4
-- 12  |  8 |  4
MySQL 8+ & SQL ServerStep 2 — Group to get island boundaries
WITH numbered AS (
  SELECT
    id,
    id - ROW_NUMBER() OVER (ORDER BY id) AS grp
  FROM numbers
)
SELECT
  MIN(id) AS island_start,
  MAX(id) AS island_end,
  COUNT(*) AS island_size
FROM numbered
GROUP BY grp
ORDER BY island_start;

-- Result:
-- island_start | island_end | island_size
-- -------------+------------+------------
-- 1            | 3          | 3
-- 6            | 7          | 2
-- 10           | 12         | 3
✅ That's the core technique

Everything else — dates, partitions, streaks — is a variation of this same pattern. Master this and the rest falls into place.

04 · Gap Queries

Finding the Gaps

Gap queries ask: "What's missing?" There are several approaches depending on how you want to express the gaps.

Pattern 1 — List Every Missing Number

MySQL 8+Find every individual missing ID
-- Using a recursive CTE to generate all numbers in range,
-- then LEFT JOIN to find which ones are missing
WITH RECURSIVE all_nums AS (
  SELECT MIN(id) AS n FROM numbers
  UNION ALL
  SELECT n + 1 FROM all_nums
  WHERE n < (SELECT MAX(id) FROM numbers)
)
SELECT a.n AS missing_id
FROM all_nums a
LEFT JOIN numbers x ON a.n = x.id
WHERE x.id IS NULL;

-- Result: 4, 5, 8, 9
SQL ServerSame using a number generator CTE
WITH all_nums AS (
  SELECT MIN(id) AS n FROM numbers
  UNION ALL
  SELECT n + 1 FROM all_nums
  WHERE n < (SELECT MAX(id) FROM numbers)
)
SELECT a.n AS missing_id
FROM all_nums a
LEFT JOIN numbers x ON a.n = x.id
WHERE x.id IS NULL
OPTION (MAXRECURSION 10000);  -- SQL Server recursion limit

Pattern 2 — List Gap Ranges (More Useful!)

Instead of listing each missing row, express gaps as start–end ranges. This uses LEAD() to peek at the next existing value.

MySQL 8+ & SQL ServerExpress gaps as ranges using LEAD()
WITH gaps AS (
  SELECT
    id                                               AS cur_val,
    LEAD(id) OVER (ORDER BY id)                   AS next_val
  FROM numbers
)
SELECT
  cur_val  + 1 AS gap_start,
  next_val - 1 AS gap_end,
  next_val - cur_val - 1 AS gap_size
FROM gaps
WHERE next_val - cur_val > 1   -- only rows where there IS a gap
ORDER BY gap_start;

-- Result:
-- gap_start | gap_end | gap_size
-- ----------+---------+---------
-- 4         | 5       | 2
-- 8         | 9       | 2
💡 How LEAD() works here

LEAD(id) gives you the value of the next row's id. If next_val - cur_val > 1, there's a gap. The gap runs from cur_val + 1 to next_val - 1.

Pattern 3 — Count Gaps / Check if Sequence is Complete

MySQL 8+ & SQL ServerQuick completeness check
-- Is the sequence perfectly complete? (no gaps at all)
SELECT
  CASE
    WHEN MAX(id) - MIN(id) + 1 = COUNT(id)
    THEN 'Complete — No Gaps!'
    ELSE 'Incomplete — Gaps exist'
  END                               AS status,
  MAX(id) - MIN(id) + 1           AS expected_count,
  COUNT(id)                         AS actual_count,
  MAX(id) - MIN(id) + 1 - COUNT(id) AS missing_count
FROM numbers;

-- Result: status='Incomplete', expected=12, actual=8, missing=4
05 · Island Queries

Finding the Islands

Method 1 — The Classic id − ROW_NUMBER()

Already introduced in Section 3. Here's the full production-ready version with per-partition support:

MySQL 8+ & SQL ServerIslands per user (partitioned)
-- employees who worked consecutive days — grouped per employee
CREATE TABLE work_log (
  emp_id   INT,
  work_date DATE
);

WITH base AS (
  SELECT
    emp_id,
    work_date,
    -- Cast date to integer for arithmetic (days since epoch)

    -- MySQL:
    TO_DAYS(work_date) -
      ROW_NUMBER() OVER (PARTITION BY emp_id ORDER BY work_date)
    AS grp
  FROM work_log
)
SELECT
  emp_id,
  MIN(work_date)  AS streak_start,
  MAX(work_date)  AS streak_end,
  COUNT(*)         AS consecutive_days
FROM base
GROUP BY emp_id, grp
ORDER BY emp_id, streak_start;
SQL ServerDate arithmetic uses DATEDIFF for integer days
WITH base AS (
  SELECT
    emp_id,
    work_date,
    DATEDIFF(DAY, '2000-01-01', work_date) -
      ROW_NUMBER() OVER (PARTITION BY emp_id ORDER BY work_date)
    AS grp
  FROM work_log
)
SELECT
  emp_id,
  MIN(work_date)  AS streak_start,
  MAX(work_date)  AS streak_end,
  COUNT(*)         AS consecutive_days
FROM base
GROUP BY emp_id, grp
ORDER BY emp_id, streak_start;

Method 2 — LAG() to Detect Break Points

This method is often cleaner and more readable. You mark the start of each new island by checking if the previous row was consecutive.

MySQL 8+ & SQL ServerMark island starts, then group with cumulative sum
WITH flagged AS (
  SELECT
    id,
    -- Mark 1 if this row starts a new island (prev row not consecutive)
    CASE
      WHEN LAG(id) OVER (ORDER BY id) = id - 1
      THEN 0   -- continuation of existing island
      ELSE   1   -- start of new island
    END AS is_start
  FROM numbers
),
grouped AS (
  SELECT
    id,
    -- Running sum of is_start = island group number
    SUM(is_start) OVER (ORDER BY id) AS island_num
  FROM flagged
)
SELECT
  island_num,
  MIN(id) AS island_start,
  MAX(id) AS island_end,
  COUNT(*) AS size
FROM grouped
GROUP BY island_num;

-- Result:
-- island_num | island_start | island_end | size
-- -----------+--------------+------------+-----
-- 1          | 1            | 3          | 3
-- 2          | 6            | 7          | 2
-- 3          | 10           | 12         | 3
🧠 How the LAG method works

Step 1: LAG(id) = id - 1 checks if the previous row is exactly one less (consecutive). If not, it's a new island start → flag it as 1.
Step 2: A running SUM of those flags gives each island an incrementing group number — a beautiful trick!

06 · Date Sequences

Gaps & Islands with Dates

Date-based problems are the most common real-world version. The only difference: you convert dates to integers (day numbers) before applying the technique.

Finding Missing Dates in a Calendar

MySQL 8+Find which dates have no sales
CREATE TABLE daily_sales (
  sale_date DATE,
  amount    DECIMAL(10,2)
);

-- Generate every date in range, then find missing ones
WITH RECURSIVE calendar AS (
  SELECT MIN(sale_date) AS d FROM daily_sales
  UNION ALL
  SELECT d + INTERVAL 1 DAY
  FROM calendar
  WHERE d < (SELECT MAX(sale_date) FROM daily_sales)
)
SELECT c.d AS missing_date
FROM calendar c
LEFT JOIN daily_sales s ON c.d = s.sale_date
WHERE s.sale_date IS NULL;
SQL ServerSame using DATEADD
WITH calendar AS (
  SELECT MIN(sale_date) AS d FROM daily_sales
  UNION ALL
  SELECT DATEADD(DAY, 1, d)
  FROM calendar
  WHERE d < (SELECT MAX(sale_date) FROM daily_sales)
)
SELECT c.d AS missing_date
FROM calendar c
LEFT JOIN daily_sales s ON c.d = s.sale_date
WHERE s.sale_date IS NULL
OPTION (MAXRECURSION 3650);  -- allow up to 10 years of dates

Finding Date Gap Ranges

MySQL 8+ & SQL ServerExpress missing date periods as ranges
WITH gaps AS (
  SELECT
    sale_date,
    LEAD(sale_date) OVER (ORDER BY sale_date) AS next_date
  FROM daily_sales
)
SELECT
  -- MySQL:
  sale_date  + INTERVAL 1 DAY   AS gap_start,
  next_date  - INTERVAL 1 DAY   AS gap_end,
  DATEDIFF(next_date, sale_date) - 1 AS missing_days  -- MySQL
  -- SQL Server: DATEDIFF(DAY, sale_date, next_date) - 1 AS missing_days
FROM gaps
WHERE DATEDIFF(next_date, sale_date) > 1  -- MySQL
ORDER BY gap_start;

Continuous Date Islands (e.g., Active Subscription Periods)

MySQL 8+Group consecutive active days into periods
CREATE TABLE subscriptions (
  user_id    INT,
  active_date DATE
);

WITH base AS (
  SELECT
    user_id,
    active_date,
    TO_DAYS(active_date) -
      ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY active_date)
    AS grp
  FROM subscriptions
)
SELECT
  user_id,
  MIN(active_date) AS period_start,
  MAX(active_date) AS period_end,
  DATEDIFF(MAX(active_date), MIN(active_date)) + 1 AS days_active
FROM base
GROUP BY user_id, grp
ORDER BY user_id, period_start;

-- Result (example):
-- user_id | period_start | period_end | days_active
-- --------+--------------+------------+------------
-- 1       | 2024-01-01   | 2024-01-15 | 15
-- 1       | 2024-02-01   | 2024-02-28 | 28   ← new period after gap
-- 2       | 2024-01-10   | 2024-01-20 | 11

Pattern A — Streak Counter (Longest Consecutive Run)

MySQL 8+ & SQL ServerFind each player's longest win streak
CREATE TABLE match_results (
  player_id   INT,
  match_date  DATE,
  result      VARCHAR(4)  -- 'WIN' or 'LOSS'
);

WITH wins_only AS (
  -- Filter to only wins first
  SELECT player_id, match_date
  FROM match_results
  WHERE result = 'WIN'
),
grouped AS (
  -- Classic island technique on the wins
  SELECT
    player_id, match_date,
    TO_DAYS(match_date) -               -- MySQL
    -- DATEDIFF(DAY,'2000-01-01',match_date) -   SQL Server
      ROW_NUMBER() OVER (PARTITION BY player_id ORDER BY match_date)
    AS grp
  FROM wins_only
),
streaks AS (
  SELECT
    player_id,
    MIN(match_date) AS streak_start,
    MAX(match_date) AS streak_end,
    COUNT(*)         AS streak_len
  FROM grouped
  GROUP BY player_id, grp
)
-- Get the longest streak per player
SELECT
  player_id,
  MAX(streak_len)  AS longest_streak
FROM streaks
GROUP BY player_id
ORDER BY longest_streak DESC;

Pattern B — Session Detection

A "session" is a group of events where no two consecutive events are more than N minutes apart. This is classic Islands — but with a time threshold instead of consecutive integers.

MySQL 8+ & SQL ServerGroup user clicks into sessions (30-min gap = new session)
CREATE TABLE user_events (
  user_id    INT,
  event_time DATETIME,
  page       VARCHAR(100)
);

WITH flagged AS (
  SELECT
    user_id, event_time, page,
    CASE
      WHEN
        -- MySQL: TIMESTAMPDIFF
        TIMESTAMPDIFF(MINUTE,
          LAG(event_time) OVER (PARTITION BY user_id ORDER BY event_time),
          event_time)
        -- SQL Server: DATEDIFF(MINUTE, LAG(...) OVER (...), event_time)
        > 30
        OR LAG(event_time) OVER (PARTITION BY user_id ORDER BY event_time) IS NULL
      THEN 1  -- new session start
      ELSE 0
    END AS is_session_start
  FROM user_events
),
sessions AS (
  SELECT
    user_id, event_time, page,
    SUM(is_session_start) OVER (PARTITION BY user_id ORDER BY event_time) AS session_id
  FROM flagged
)
SELECT
  user_id,
  session_id,
  MIN(event_time) AS session_start,
  MAX(event_time) AS session_end,
  COUNT(*)          AS event_count
FROM sessions
GROUP BY user_id, session_id
ORDER BY user_id, session_start;

Pattern C — Merging Overlapping Date Ranges

You have overlapping intervals (e.g., hotel bookings, employee contracts) and want to merge them into non-overlapping continuous ranges.

MySQL 8+ & SQL ServerMerge overlapping project assignments
CREATE TABLE assignments (
  emp_id     INT,
  start_date DATE,
  end_date   DATE
);
-- Sample: emp 1 has [Jan1-Jan10], [Jan5-Jan20], [Feb1-Feb15]
-- Expected merge: [Jan1-Jan20], [Feb1-Feb15]

WITH ordered AS (
  SELECT
    emp_id, start_date, end_date,
    -- Max end_date seen so far among all previous ranges
    MAX(end_date) OVER (
      PARTITION BY emp_id
      ORDER BY start_date
      ROWS BETWEEN UNBOUNDED PRECEDING AND 1 PRECEDING
    ) AS prev_max_end
  FROM assignments
),
flagged AS (
  SELECT
    emp_id, start_date, end_date,
    -- New group if start is after all previous ends (no overlap)
    SUM(CASE WHEN start_date <= prev_max_end THEN 0 ELSE 1 END)
      OVER (PARTITION BY emp_id ORDER BY start_date) AS grp
  FROM ordered
)
SELECT
  emp_id,
  MIN(start_date) AS merged_start,
  MAX(end_date)   AS merged_end
FROM flagged
GROUP BY emp_id, grp
ORDER BY emp_id, merged_start;

Pattern D — Find the Next Available (First Missing) Number

MySQL 8+ & SQL ServerFind the smallest unused ID
-- Use case: assign the next available ticket number, invoice ID, etc.
SELECT MIN(t1.id) + 1 AS next_available_id
FROM numbers t1
WHERE NOT EXISTS (
  SELECT 1 FROM numbers t2
  WHERE t2.id = t1.id + 1
);

-- Alternative: cleaner with LAG
WITH gaps AS (
  SELECT
    id,
    LEAD(id) OVER (ORDER BY id) AS next_id
  FROM numbers
)
SELECT MIN(id + 1) AS first_gap
FROM gaps
WHERE next_id > id + 1 OR next_id IS NULL;

Pattern E — State Change Detection

When a column's value changes (e.g., a machine switches from ON to OFF), detect the periods of each state.

MySQL 8+ & SQL ServerTrack machine uptime/downtime periods
CREATE TABLE machine_status (
  machine_id  INT,
  ts          DATETIME,
  status      VARCHAR(10)  -- 'UP' or 'DOWN'
);

WITH state_groups AS (
  SELECT
    machine_id, ts, status,
    -- New group whenever the status CHANGES from the previous row
    SUM(
      CASE
        WHEN status = LAG(status) OVER (PARTITION BY machine_id ORDER BY ts)
        THEN 0
        ELSE 1
      END
    ) OVER (PARTITION BY machine_id ORDER BY ts) AS grp
  FROM machine_status
)
SELECT
  machine_id,
  status,
  MIN(ts) AS period_start,
  MAX(ts) AS period_end
FROM state_groups
GROUP BY machine_id, status, grp
ORDER BY machine_id, period_start;
💡 Key Insight for State Changes

When status = LAG(status) the state hasn't changed → same group (0). When they differ → new group (1). The running SUM() bumps up the group number each time a change occurs.

Healthcare — Continuous Hospital Stays
Merge daily admission records into full stay periods
MySQL 8+Detect continuous hospitalisation periods
CREATE TABLE admissions (
  patient_id  INT,
  stay_date   DATE
);

WITH base AS (
  SELECT
    patient_id, stay_date,
    TO_DAYS(stay_date) -
      ROW_NUMBER() OVER (PARTITION BY patient_id ORDER BY stay_date) AS grp
  FROM admissions
)
SELECT
  patient_id,
  MIN(stay_date)                                         AS admitted_on,
  MAX(stay_date)                                         AS discharged_on,
  DATEDIFF(MAX(stay_date), MIN(stay_date)) + 1          AS length_of_stay
FROM base
GROUP BY patient_id, grp
HAVING length_of_stay >= 3   -- only stays of 3+ days
ORDER BY patient_id;

Finance — Continuous Stock Price Rises
Find periods where price rose every single day
SQL ServerDetect consecutive up-days in stock prices
CREATE TABLE stock_prices (
  ticker     VARCHAR(10),
  price_date DATE,
  close_price DECIMAL(10,2)
);

WITH daily_change AS (
  SELECT
    ticker, price_date, close_price,
    LAG(close_price) OVER (PARTITION BY ticker ORDER BY price_date) AS prev_price
  FROM stock_prices
),
up_days AS (
  -- Keep only days where price went UP
  SELECT ticker, price_date, close_price
  FROM daily_change
  WHERE close_price > prev_price
),
grouped AS (
  SELECT
    ticker, price_date, close_price,
    DATEDIFF(DAY, '2000-01-01', price_date) -
      ROW_NUMBER() OVER (PARTITION BY ticker ORDER BY price_date) AS grp
  FROM up_days
)
SELECT
  ticker,
  MIN(price_date) AS rally_start,
  MAX(price_date) AS rally_end,
  COUNT(*)          AS consecutive_up_days
FROM grouped
GROUP BY ticker, grp
HAVING COUNT(*) >= 5    -- only rallies of 5+ days
ORDER BY consecutive_up_days DESC;

E-Commerce — Finding Inactive Customer Periods
When did a customer stop buying, and for how long?
MySQL 8+Detect customer churn gaps (months with no purchase)
CREATE TABLE orders (
  customer_id INT,
  order_date  DATE
);

WITH monthly AS (
  -- Collapse to distinct purchase months per customer
  SELECT DISTINCT
    customer_id,
    DATE_FORMAT(order_date, '%Y-%m-01') AS month_start
  FROM orders
),
with_gap AS (
  SELECT
    customer_id,
    month_start,
    LEAD(month_start) OVER (PARTITION BY customer_id ORDER BY month_start) AS next_purchase
  FROM monthly
)
SELECT
  customer_id,
  month_start                  AS last_purchase_month,
  next_purchase                AS returned_month,
  TIMESTAMPDIFF(MONTH, month_start, next_purchase) - 1 AS inactive_months
FROM with_gap
WHERE
  next_purchase IS NOT NULL
  AND TIMESTAMPDIFF(MONTH, month_start, next_purchase) > 1
ORDER BY inactive_months DESC;

CREATE TABLE logins (
  player_id  INT,
  login_date DATE
);

WITH base AS (
  SELECT DISTINCT player_id, login_date FROM logins
),
grouped AS (
  SELECT
    player_id, login_date,

  FROM base

FROM streaks
GROUP BY player_id
ORDER BY all_time_best DESC;
09 · Interview Prep

Interview Questions

Easy
What is the "Gaps and Islands" problem in SQL?
Gaps and Islands is a category of SQL problems dealing with sequences — usually numbers or dates. A gap is a missing value in a sequence (e.g., employee IDs 1,2,3,7,8 — IDs 4,5,6 are a gap). An island is a group of consecutive existing values (e.g., [1–3] and [7–8] are two islands). The classic solution uses ROW_NUMBER() and the mathematical property that value − row_number is constant within a consecutive group.
Easy
Why does value − ROW_NUMBER() identify islands?
If a sequence is consecutive (1,2,3), the row numbers are also consecutive (1,2,3), so the difference is always 0. If there's a gap — say the next value jumps to 6 — but the row number only increments by 1 (to 4), the difference jumps to 2. So every island has a constant difference, and that constant uniquely identifies the island. Grouping by this difference gives you each island as a distinct group.
Medium
How would you find date gaps — days where no sales occurred?
Two approaches: (1) Calendar generation — use a recursive CTE to generate every date in the range, then LEFT JOIN your sales table and keep rows where the join returns NULL (no matching sale). (2) LEAD() approach — use LEAD(sale_date) OVER (ORDER BY sale_date) to get the next date, then filter rows where next_date - sale_date > 1. This gives you gap ranges without generating a calendar. Approach 2 is more efficient for large datasets.
Medium
What is the LAG/SUM technique for detecting islands, and how does it differ from id − ROW_NUMBER()?
The LAG method: use LAG(value) to check if the previous row is consecutive. If not, flag it as 1 (new island start), else 0. Then take a running SUM() of those flags — this creates an incrementing group counter that bumps up at each island boundary. Difference from id−RN: the LAG/SUM method works even if values aren't simple integers (e.g., it generalises better to custom "consecutive" definitions like time-based gaps). The id−RN method is simpler and more common for pure integer/date sequences.
Medium
How do you handle the Gaps & Islands problem when data is partitioned (e.g., per user)?
Add PARTITION BY user_id to the ROW_NUMBER() window function. This resets the row number to 1 for each user, so the value − row_number difference is computed independently per user. Example: ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY date). Then also include user_id in the final GROUP BY so each user's islands are separate.
Medium
How would you find the longest consecutive sequence (e.g., longest login streak)?
Apply the island technique — filter to only the relevant rows (e.g., login dates), compute groups using date − ROW_NUMBER() partitioned by user, then GROUP BY user, grp and COUNT(*) to get each streak's length. Finally, take MAX(streak_length) per user. The key insight is: filter to only the "qualifying" rows first (wins, logins, up-days), then apply the island technique — the gaps in the original data naturally separate the islands.
Medium
What is session detection, and how is it a Gaps & Islands problem?
Session detection groups user events where no two consecutive events are more than N minutes apart (typically 30 minutes). It's an islands problem because you're grouping "consecutive" events — but "consecutive" here means within a time threshold, not exactly adjacent numbers. Solution: use LAG(event_time) to compute the gap to the previous event. Flag a new session when the gap exceeds the threshold. Then use a running SUM() of those flags as a session ID. The session ID increments each time a new session starts.
Hard
How do you merge overlapping date ranges into non-overlapping continuous periods?
This is the hardest Gaps & Islands variant. The key is to detect when a new range's start date falls after the maximum end date of all previous ranges (meaning no overlap → new group). Use MAX(end_date) OVER (... ROWS BETWEEN UNBOUNDED PRECEDING AND 1 PRECEDING) to compute the running maximum end date. Then: if start_date ≤ prev_max_end, it overlaps with an existing period (same group = 0); otherwise it starts a new period (= 1). A running SUM gives the group number. Finally GROUP BY and take MIN(start) and MAX(end) per group.
Hard
How would you find gaps AND islands in a single query efficiently?
Use LEAD() to compute both simultaneously. For each row: LEAD(id) - id - 1 gives the gap size after this island ends. Rows where this value > 0 are island endpoints that are followed by a gap. You can join or union the island result (from the ROW_NUMBER grouping) with gap results (from LEAD filtering) in a single CTE chain. Alternatively, compute the islands first, then derive gaps as the spaces between consecutive islands: next_island_start - 1 is the gap end, prev_island_end + 1 is the gap start.
Hard
Does the id − ROW_NUMBER() technique work for dates directly? Why or why not?
Not directly — you can't subtract a row number (integer) from a DATE type. You first need to convert the date to an integer representing the number of days since a fixed epoch. In MySQL, use TO_DAYS(date). In SQL Server, use DATEDIFF(DAY, '2000-01-01', date). Once both are integers, the subtraction works identically. The result will be an integer that is constant within a consecutive daily island. Important: this only works when "consecutive" means one calendar day apart — for business days only, you'd need a different approach (e.g., a business day calendar table).
Hard
Write a query to detect state changes — group a machine's status log into continuous periods per state.
Use LAG(status) to compare each row's status to the previous one. When they differ, flag it as 1 (new state period begins). A running SUM(flag) OVER (PARTITION BY machine_id ORDER BY timestamp) gives an incrementing group number that advances whenever the state changes. Then GROUP BY machine_id, status, grp and take MIN(ts) and MAX(ts) for each period. Note: include status in both SELECT and GROUP BY — not just grp — because consecutive periods of the same state get different grp values (e.g., if a machine goes UP→DOWN→UP, the two UP periods are separate groups).
Medium
What are the performance considerations for Gaps & Islands queries on large tables?
Key optimisations: (1) Indexes — always have an index on the sequence column (date or ID) used in ORDER BY of window functions. For partitioned queries, a composite index on (partition_col, sequence_col) is ideal. (2) Filter first — apply WHERE conditions before the window functions using CTEs, so the window operates on fewer rows. (3) Avoid recursive CTEs for large ranges — generating a full calendar via recursion can be slow; prefer a pre-built calendar/numbers table. (4) Materialise CTEs — in SQL Server, break complex chains into temp tables to allow the query optimizer to use statistics between steps.

The Cheatsheet

🏝️ Islands — Core Pattern
  • Step 1: Add ROW_NUMBER() OVER (ORDER BY col)
  • Step 2: Compute col - row_number as grp
  • Step 3: GROUP BY grp
  • Step 4: MIN(col) = start, MAX(col) = end
  • Partitioned? Add PARTITION BY in ROW_NUMBER()
  • Dates? Convert to int first (TO_DAYS / DATEDIFF)
🕳️ Gaps — Core Pattern
  • Range gaps: Use LEAD(col) - col > 1
  • gap_start = current + 1
  • gap_end = next_val - 1
  • Individual missing: Generate range + LEFT JOIN
  • Quick check: MAX - MIN + 1 = COUNT (no gaps)
  • First missing: Find min id where next ≠ id+1
⚡ Window Functions Used
  • ROW_NUMBER() Core of islands technique
  • LAG(col) Look at previous row's value
  • LEAD(col) Look at next row's value
  • SUM() OVER Running total for group IDs
  • MAX() OVER Running max for overlap merging
  • PARTITION BY Reset per user/category
🗓️ Date Arithmetic
  • MySQL integer: TO_DAYS(date_col)
  • SQL Server int: DATEDIFF(DAY,'2000-01-01',col)
  • MySQL next day: date + INTERVAL 1 DAY
  • SQL Server next: DATEADD(DAY, 1, date)
  • MySQL diff: DATEDIFF(d2, d1)
  • SS diff: DATEDIFF(DAY, d1, d2)
🎯 Pattern Lookup
  • Login streaks: Filter wins → island technique
  • User sessions: LAG time gap > threshold
  • State changes: LAG(status) ≠ status → new group
  • Overlap merge: Running MAX of end dates
  • Churn gaps: LEAD on monthly purchases
  • Uptime periods: Island on 'UP' rows only
🚀 Performance Tips
  • Index: (partition_col, sequence_col)
  • Filter first in a CTE before windowing
  • Avoid recursion on huge ranges → calendar table
  • SQL Server: temp tables between CTE steps
  • SELECT DISTINCT before island logic for dupes
  • OPTION(MAXRECURSION N) for deep SS recursion

WITH g AS (
  SELECT col,
    col - ROW_NUMBER() OVER(ORDER BY col) AS grp     -- integers
    -- TO_DAYS(col) - ROW_NUMBER()...                    -- MySQL dates
    -- DATEDIFF(DAY,'2000-01-01',col) - ROW_NUMBER()...  -- SS dates
  FROM tbl
)
SELECT MIN(col) start_, MAX(col) end_, COUNT(*) size_
FROM g GROUP BY grp;

FROM g WHERE nxt - col > 1;

WITH f AS (
  SELECT user_id, ts,
    SUM(CASE WHEN TIMESTAMPDIFF(MINUTE, LAG(ts)OVER(PARTITION BY user_id ORDER BY ts), ts) > 30
                  OR LAG(ts)OVER(...) IS NULL THEN 1 ELSE 0 END)
    OVER(PARTITION BY user_id ORDER BY ts) session_id
  FROM events
)
SELECT user_id, session_id, MIN(ts) start_, MAX(ts) end_
FROM f GROUP BY user_id, session_id;

━━━ STATE CHANGES ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
WITH sg AS (
  SELECT id, ts, state,
    SUM(CASE WHEN state = LAG(state)OVER(PARTITION BY id ORDER BY ts)
              THEN 0 ELSE 1 END)
    OVER(PARTITION BY id ORDER BY ts) grp
  FROM status_log
)
SELECT id, state, MIN(ts) start_, MAX(ts) end_
FROM sg GROUP BY id, state, grp;
