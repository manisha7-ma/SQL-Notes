01 · Foundation

What is an Index?

An index is like the index at the back of a textbook. Instead of reading every page to find "photosynthesis", you flip to the index, see "page 247", and jump directly there. Without it, you'd scan every single page — that's a full table scan.

A database index is a separate data structure that stores a sorted copy of selected column values alongside pointers to the actual row locations on disk. This makes finding rows dramatically faster — at the cost of extra storage and slightly slower writes.

Without an Index

SQLFull Table Scan — O(n)
-- Table: users (10 million rows)
SELECT * FROM users WHERE email = 'alice@example.com';
-- 🐌 DB scans ALL 10 million rows one-by-one
-- Time: ~8 seconds on spinning disk

With an Index

SQLIndex Lookup — O(log n)
-- Create index on email column
CREATE INDEX idx_users_email ON users(email);

SELECT * FROM users WHERE email = 'alice@example.com';
-- ⚡ DB traverses B-tree: ~23 comparisons (log₂ 10M)
-- Time: ~0.1ms
✅ The Payoff

10,000,000 rows → ~23 comparisons instead of 10,000,000. That's a 400,000× speedup in the best case.

02 · Internals

How Indexes Work

Most SQL databases use a B-Tree (Balanced Tree) as the default index structure. Here's what that means:

DiagramB-Tree Structure for idx_users_email
                    [Root Node]
                    ["m@.." | "r@.."]
                   /        |        \
          [Leaf A]       [Leaf B]       [Leaf C]
       a@.. → row3    m@.. → row7    r@.. → row1
       c@.. → row9    p@.. → row2    z@.. → row5
       k@.. → row4    q@.. → row8

Each leaf holds:  (index_key_value → physical row pointer)

Step-by-Step Lookup

ProcessFinding email = 'p@example.com'
Step 1: Read Root Node  → "p" > "m" AND "p" < "r"  → go to Leaf B
Step 2: Read Leaf B     → found "p@.." → pointer = row2
Step 3: Fetch row2 from disk

Total disk reads: 3  (vs 10,000,000 for full scan)

03 · Taxonomy

Types of Indexes

🔑
Primary Index
Auto-created on PRIMARY KEY. Rows are physically stored in this order. Only one per table.
🔗
Unique Index
Enforces uniqueness + speeds up lookups. Created implicitly by UNIQUE constraint.
📋
Non-Unique Index
Standard index on any column. Speeds up queries, allows duplicate values.
🧩
Composite Index
Index on 2+ columns. Column order matters! Left-prefix rule applies.
📦
Covering Index
Includes all columns a query needs — eliminates table lookups entirely.
🔍
Full-Text Index
For MATCH...AGAINST searches on text content. Not a simple B-tree.
🌐
Spatial Index
For GIS / geographic data. Uses R-Tree structure for location queries.
⚙️
Filtered / Partial
Index only a subset of rows using a WHERE clause. Smaller & faster.

1. Primary Index (Clustered)

SQLCreated automatically with PRIMARY KEY
CREATE TABLE orders (
  order_id   INT          NOT NULL AUTO_INCREMENT,
  user_id    INT          NOT NULL,
  total      DECIMAL(10,2),
  created_at DATETIME,
  PRIMARY KEY (order_id)  -- Auto-creates clustered index
);
-- Rows on disk are physically sorted by order_id
-- Range scan on order_id is extremely fast

2. Unique Index

SQLEnforce uniqueness + performance
-- Option A: via constraint
ALTER TABLE users ADD CONSTRAINT uq_email UNIQUE(email);

-- Option B: explicit unique index
CREATE UNIQUE INDEX idx_users_email ON users(email);

-- Allows NULLs (multiple NULLs permitted in most DBs)
-- Duplicate insert → ERROR: Duplicate entry

3. Regular (Non-Unique) Index

SQLSpeed up frequent lookups
-- Index on foreign key column (very common pattern)
CREATE INDEX idx_orders_user_id ON orders(user_id);

-- Now this JOIN is fast:
SELECT o.* FROM orders o
JOIN users u ON o.user_id = u.id
WHERE u.id = 42;
-- Without idx_orders_user_id: full scan of orders table!

4. Partial / Filtered Index

PostgreSQLIndex only relevant rows — smaller & faster
-- Use case: 95% of orders are 'completed', queries only hit 'pending'
CREATE INDEX idx_pending_orders ON orders(created_at)
WHERE status = 'pending';

-- This query hits the partial index (fast + small index):
SELECT * FROM orders
WHERE status = 'pending'
  AND created_at > NOW() - INTERVAL '7 days';

-- Index size: maybe 5% of a full index on the same column

5. Full-Text Index

MySQLSearch within text content
ALTER TABLE articles
ADD FULLTEXT INDEX ft_article_content(title, body);

-- Natural language search
SELECT *, MATCH(title, body) AGAINST('database optimization') AS relevance
FROM articles
WHERE MATCH(title, body) AGAINST('database optimization' IN NATURAL LANGUAGE MODE)
ORDER BY relevance DESC;

-- Boolean mode with operators
WHERE MATCH(title, body) AGAINST('+database -nosql optimization*' IN BOOLEAN MODE);
04 · Syntax Reference

Create, Drop & Manage

Creating Indexes

SQLAll creation patterns
-- Basic index
CREATE INDEX idx_name ON table_name(column_name);

-- Unique index
CREATE UNIQUE INDEX idx_name ON table_name(column_name);

-- Composite index (column order matters!)
CREATE INDEX idx_name ON table_name(col1, col2, col3);

-- Inline during table creation
CREATE TABLE products (
  id         INT PRIMARY KEY,
  sku        VARCHAR(50) UNIQUE,
  category   VARCHAR(50),
  price      DECIMAL(10,2),
  INDEX idx_category(category),      -- inline index
  INDEX idx_cat_price(category, price) -- composite inline
);

-- PostgreSQL: CONCURRENTLY (no table lock!)
CREATE INDEX CONCURRENTLY idx_users_city ON users(city);
-- Takes longer but doesn't block reads/writes. Use in production!

Dropping Indexes

SQLRemove indexes
-- MySQL
DROP INDEX idx_users_email ON users;
ALTER TABLE users DROP INDEX idx_users_email;

-- PostgreSQL / SQL Server
DROP INDEX idx_users_email;
DROP INDEX IF EXISTS idx_users_email; -- safe version

Viewing Existing Indexes

SQLInspect indexes
-- MySQL
SHOW INDEXES FROM users;
SHOW INDEX FROM orders;

-- PostgreSQL
SELECT indexname, indexdef
FROM pg_indexes
WHERE tablename = 'users';

-- SQL Server
SELECT name, type_desc, is_unique
FROM sys.indexes
WHERE object_id = OBJECT_ID('users');
05 · Strategy

When to Use Indexes

🎯 The Rule of Thumb

Index columns that appear in WHERE, JOIN ON, ORDER BY, or GROUP BY clauses — and only if those queries run frequently.

✅ Good Candidates for Indexing

Use Index

High-Cardinality WHERE Columns

SQLemail, user_id, order_id — many unique values
-- email has millions of unique values → index is very selective
CREATE INDEX idx_users_email ON users(email);

SELECT * FROM users WHERE email = 'alice@example.com';
-- Without index: 10M rows scanned
-- With index:    1 row found via B-tree in ~23 steps
Use Index

Foreign Keys (JOIN Columns)

SQLAlways index FK columns!
CREATE TABLE comments (
  id      INT PRIMARY KEY,
  post_id INT NOT NULL,     -- FK
  user_id INT NOT NULL,     -- FK
  body    TEXT
);

-- Without these, every JOIN does a full scan on comments!
CREATE INDEX idx_comments_post_id ON comments(post_id);
CREATE INDEX idx_comments_user_id ON comments(user_id);

-- This JOIN is now fast:
SELECT c.body FROM comments c
JOIN posts p ON c.post_id = p.id
WHERE p.id = 1234;
Use Index

ORDER BY + LIMIT Queries

SQLAvoid sorting millions of rows
-- Without index: sort ALL orders, then take top 10 → expensive
-- With index: already sorted, just read top 10 → instant
CREATE INDEX idx_orders_created_at ON orders(created_at DESC);

SELECT * FROM orders
ORDER BY created_at DESC
LIMIT 10;
-- Index provides pre-sorted data → no filesort → O(1) effective
Use Index

Range Queries

SQLBETWEEN, >, <, date ranges
CREATE INDEX idx_orders_total ON orders(total);

-- Index enables fast range scan:
SELECT * FROM orders
WHERE total BETWEEN 100 AND 500;

-- Same with dates:
SELECT * FROM orders
WHERE created_at >= '2024-01-01'
  AND created_at < '2024-02-01';
-- B-tree finds start point, scans forward → very efficient
06 · Pitfalls

When Indexes Don't Work

⚠️ The Classic Mistakes

These patterns silently kill your index. The query runs, but the DB does a full scan anyway.

1. Function on Indexed Column

SQLIndex on created_at is IGNORED here
-- ❌ BAD: wrapping a column in a function defeats the index
SELECT * FROM orders
WHERE YEAR(created_at) = 2024;
-- DB must evaluate YEAR() for every row → full scan!

-- ✅ GOOD: range query preserves index usage
SELECT * FROM orders
WHERE created_at >= '2024-01-01'
  AND created_at < '2025-01-01';
-- Index on created_at is used ✓

2. Leading Wildcard in LIKE

SQL% at start = full scan
-- ❌ BAD: leading wildcard can't use B-tree
SELECT * FROM users WHERE name LIKE '%alice%'; -- full scan
SELECT * FROM users WHERE name LIKE '%alice';  -- full scan

-- ✅ GOOD: trailing wildcard can use index
SELECT * FROM users WHERE name LIKE 'alice%';  -- uses index!

-- ✅ BETTER: for %word% searches, use FULLTEXT index
MATCH(name) AGAINST('alice')

3. Implicit Type Conversion

SQLMismatched types bypass indexes
-- Table: users.phone is VARCHAR(20), indexed

-- ❌ BAD: comparing VARCHAR to INT triggers implicit cast
SELECT * FROM users WHERE phone = 9876543210; -- full scan!

-- ✅ GOOD: match the column type exactly
SELECT * FROM users WHERE phone = '9876543210'; -- index used!

4. OR Can Defeat Indexes

SQLOR often causes full scan
-- ❌ May not use index (DB-dependent)
SELECT * FROM users
WHERE email = 'a@b.com' OR phone = '123';

-- ✅ BETTER: UNION to force each index path
SELECT * FROM users WHERE email = 'a@b.com'
UNION
SELECT * FROM users WHERE phone = '123';
-- Each branch can independently use its index

5. Low Cardinality Columns

SQLWaste of an index
-- ❌ BAD: boolean/status with 2-3 values → index not selective
CREATE INDEX idx_is_active ON users(is_active);
-- is_active is TRUE/FALSE → 50% of rows match → DB prefers full scan

-- ✅ BETTER: partial index (only index the minority case)
CREATE INDEX idx_inactive_users ON users(id)
WHERE is_active = FALSE;
-- Small index, only for the rare case (0.1% of users maybe)

6. NOT IN / NOT EQUAL

SQLNegations are hard to index
-- ❌ BAD: negations often lead to full scans
SELECT * FROM orders WHERE status != 'cancelled';
SELECT * FROM orders WHERE status NOT IN ('cancelled', 'refunded');

-- ✅ REWRITE: if the positive set is small
SELECT * FROM orders
WHERE status IN ('pending', 'processing', 'shipped');
07 · Composite Indexes

The Column Order Rule

A composite index on (last_name, first_name) is like a phone book. You can quickly find all "Smiths", or "Smith, Alice" — but you can't jump directly to all "Alice"s without scanning every entry.

A composite index on (A, B, C) can be used for queries on A, (A, B), or (A, B, C) — but NOT on just B or C alone. This is the Left-Prefix Rule.

SQLLeft-Prefix Rule in detail
CREATE INDEX idx_orders_composite
ON orders(user_id, status, created_at);

-- ✅ Uses index (starts from left)
WHERE user_id = 5
WHERE user_id = 5 AND status = 'pending'
WHERE user_id = 5 AND status = 'pending' AND created_at > '2024-01-01'

-- ❌ Does NOT use index (skips left columns)
WHERE status = 'pending'
WHERE created_at > '2024-01-01'
WHERE status = 'pending' AND created_at > '2024-01-01'

-- ⚠️ Partial use: uses index on user_id only
WHERE user_id = 5 AND created_at > '2024-01-01'
-- status is skipped → index used for user_id, then filtered

Ordering Columns in Composite Index

SQLEquality columns first, range columns last
-- RULE: Put equality (=) columns before range (>, <, BETWEEN)

-- Query pattern:
SELECT * FROM orders
WHERE user_id = 5              -- equality
  AND status  = 'shipped'     -- equality
  AND created_at > '2024-01-01'; -- range

-- ✅ CORRECT order: equality cols first, range col last
CREATE INDEX idx_good ON orders(user_id, status, created_at);

-- ❌ BAD order: range col in middle stops index use
CREATE INDEX idx_bad ON orders(user_id, created_at, status);
-- Index used for (user_id, created_at) only — status not indexed

Real-World Composite Example: E-commerce

SQLMulti-tenant SaaS pattern
-- Common query in a multi-tenant app:
SELECT * FROM transactions
WHERE tenant_id = 42
  AND account_id = 1001
  AND txn_date BETWEEN '2024-01-01' AND '2024-03-31'
ORDER BY txn_date DESC;

-- Perfect composite index:
CREATE INDEX idx_txn_tenant_account_date
ON transactions(tenant_id, account_id, txn_date);
-- 1. Jumps to tenant 42
-- 2. Within that, to account 1001
-- 3. Within that, range scan on txn_date
-- 4. Already sorted for ORDER BY — no filesort!
08 · Covering Indexes

The Index-Only Scan

A covering index is like a cheat sheet with all the answers. The DB doesn't even need to open the main table — everything needed is right there in the index itself.

SQLEliminating the table lookup
-- Query:
SELECT user_id, status, total
FROM orders
WHERE user_id = 42
  AND status = 'completed';

-- Non-covering index — two operations:
-- 1) Find rows in idx_orders_user_id → get row pointers
-- 2) Fetch each row from the TABLE to get 'total'
CREATE INDEX idx_orders_user_id ON orders(user_id);

-- Covering index — one operation (index has everything!):
CREATE INDEX idx_orders_covering
ON orders(user_id, status, total);
-- All 3 SELECTed columns + WHERE columns are IN the index
-- EXPLAIN shows: "Using index" (no table access!) ✓
PostgreSQLINCLUDE for non-key covering columns
-- PostgreSQL 11+: INCLUDE adds columns without affecting key ordering
CREATE INDEX idx_orders_covering_pg
ON orders(user_id, status)
INCLUDE (total, created_at);
-- user_id, status = searchable key columns
-- total, created_at = included for retrieval only (not sorted)
-- Index is slightly smaller than indexing all 4 as keys
⚖️ Trade-off

Covering indexes are larger on disk. Don't cover 10 columns "just in case" — only add covering columns for high-frequency, performance-critical queries.

SQLIndexing strategy for an online store
CREATE TABLE products (

  ON products(category_id, price);

CREATE INDEX idx_products_brand_active
  ON products(brand_id, is_active);

-- 3. Price range filter (common in search)
CREATE INDEX idx_products_price
  ON products(price);

-- 4. Full-text search on name/description
ALTER TABLE products
  ADD FULLTEXT ft_products_search(name, description);

-- 5. Low-stock alerts (partial index — only relevant rows)
CREATE INDEX idx_low_stock ON products(stock)
  WHERE stock < 10 AND is_active = TRUE;

SELECT p.id, p.content, p.created_at, u.name

Analytics / Reporting

SQLAggregation with indexes
-- Monthly revenue report
SELECT
  DATE_FORMAT(created_at, '%Y-%m') AS month,
  COUNT(*) AS order_count,
  SUM(total) AS revenue
FROM orders
WHERE created_at BETWEEN '2024-01-01' AND '2024-12-31'

-- Covering index for this exact report:
CREATE INDEX idx_orders_report_covering
  ON orders(status, created_at)

-- status (=) first, created_at (range) second, total included
-- Entire query served from index — no table touch!

Benefits & Trade-offs Summary

✅ Benefits
  • Dramatically faster SELECT queries
  • Enables efficient JOINs on large tables
  • Eliminates expensive filesorts (ORDER BY)
  • Enforces uniqueness (UNIQUE index)
  • Reduces I/O — fewer disk reads
  • Covering indexes avoid table access entirely
❌ Costs & Risks
  • Extra disk space for each index
  • Slower INSERT, UPDATE, DELETE (index must update)
  • DB must choose the right index (can be wrong)
  • Too many indexes → optimizer confusion
  • Over-indexing write-heavy tables kills throughput
  • Unused indexes waste space and slow writes
11 · Interview Prep

Interview Questions

What is an index in SQL and why do we use it?
An index is a separate data structure (usually a B-tree) that stores sorted column values with pointers to the actual rows. We use indexes to speed up data retrieval — a query that would scan millions of rows becomes a logarithmic-time lookup. The trade-off is extra storage and slower write operations, since the index must be updated on every INSERT, UPDATE, or DELETE.
What's the difference between a clustered and non-clustered index?
A clustered index determines the physical order of data on disk — rows are stored sorted by the indexed column. There can only be one per table (usually the primary key). A non-clustered index is a separate structure with pointers back to the heap/data pages. You can have many non-clustered indexes per table. In MySQL InnoDB, the primary key is always clustered; all other indexes are non-clustered and store the primary key value as their row pointer.
Explain the left-prefix rule for composite indexes.
For a composite index on (A, B, C), the DB can use the index for queries filtering on A alone, A+B together, or A+B+C. But it cannot efficiently use the index for queries on B alone, C alone, or B+C — because the B-tree is sorted first by A. Think of it like a phone book sorted by (last_name, first_name): you can find all Smiths, or Smith+Alice — but not all "Alice"s without scanning every entry.
What is a covering index?
A covering index is one where all the columns needed by a query (both in SELECT and WHERE) are present in the index itself — so the database can answer the query entirely from the index without touching the main table. In EXPLAIN output, you'll see "Using index". This is the highest level of index optimization because it eliminates the expensive random I/O of fetching actual rows.
When would you NOT use an index?
Several cases: (1) Low-cardinality columns like boolean flags — if 50% of rows match, a full scan is often cheaper. (2) Small tables where a full scan is trivial. (3) Write-heavy tables like logging tables where indexes slow down every INSERT. (4) Columns rarely used in WHERE/JOIN/ORDER BY clauses. (5) When the column has many NULLs and you rarely query non-NULL values.
List 4 situations where an index won't be used.
(1) Using a function on the indexed column: WHERE YEAR(date) = 2024. (2) A leading wildcard in LIKE: WHERE name LIKE '%alice'. (3) Implicit type conversion: comparing a VARCHAR column to an integer literal. (4) Negations like NOT IN or != — the DB often determines a full scan is cheaper. Also: (5) Low selectivity — if the query matches >20-30% of rows, the query planner may skip the index entirely.
How does adding an index affect write performance?
Every index on a table must be updated on INSERT, UPDATE, and DELETE. A table with 5 indexes requires 6 write operations for every row insert (1 data + 5 index updates). This makes indexes costly on write-heavy tables. Strategies: use partial indexes (index only relevant subsets), drop unused indexes, or use INSERT IGNORE batching. On bulk loads, it's often faster to drop indexes first, load data, then rebuild them.
What is a partial index and when would you use one?
A partial index (PostgreSQL) only indexes rows that satisfy a WHERE condition in the index definition — e.g., CREATE INDEX ON orders(created_at) WHERE status = 'pending'. Use it when: (1) most queries target a minority of rows (like only active or pending records), (2) the index would otherwise be large and most of it unused, (3) you want to enforce a conditional uniqueness constraint (e.g., unique email among active users only).
How do you check if a query is using an index?
Use EXPLAIN (MySQL) or EXPLAIN ANALYZE (PostgreSQL). In MySQL, look at the type column — ref, range, or const means an index is used; ALL means a full table scan. The key column shows which index was chosen, and rows shows estimated rows scanned. In PostgreSQL, look for "Index Scan" or "Index Only Scan" (best) vs "Seq Scan" (worst).
What is index cardinality and why does it matter?
Cardinality is the number of unique values in an indexed column. High cardinality (e.g., email, UUID) = very selective index = few rows returned per lookup = very efficient. Low cardinality (e.g., boolean, gender, status) = less selective = many rows returned per lookup = index may be skipped. The query optimizer uses cardinality statistics to decide whether to use an index — if an index returns 40% of rows, a sequential scan is often faster.
Explain the difference between B-Tree and Hash indexes.
A B-Tree index stores keys in a sorted balanced tree structure — it supports equality (=), range (>, <, BETWEEN), prefix (LIKE 'abc%'), and ORDER BY operations. It's the default and most versatile. A Hash index uses a hash function — it only supports exact equality lookups (=) in O(1) time, but cannot do range queries or sorting. Use Hash for simple key-value lookup tables; B-Tree for everything else.
12 · Quick Reference

The Cheatsheet

✅ Always Index These
  • PRIMARY KEY — auto-created
  • FOREIGN KEYS — join performance
  • Unique constraints — auto-created
  • Frequent WHERE columns — high cardinality
  • ORDER BY + LIMIT columns — avoid filesort
  • JOIN ON columns — both sides
❌ Avoid Indexing These
  • Boolean / status — low cardinality
  • Rarely queried columns — waste
  • Large text blobs — use fulltext
  • Write-only tables — overhead
  • Columns in functions — index ignored
  • Tiny tables — full scan faster
🧩 Composite Index Rules
  • Equality (=) columns go first
  • Range (>, <, BETWEEN) go last
  • ORDER BY column goes last if possible
  • Left-prefix rule must be satisfied
  • Most selective column usually first
  • Max ~3-4 columns per index
🔍 EXPLAIN Signals
  • type=ALL → add an index
  • Using filesort → index ORDER BY
  • Using temporary → restructure query
  • Using index → covering index (best!)
  • rows very high → index not selective
  • key=NULL → no index used
💡 Index Killers
  • YEAR(date_col) → use date range
  • LIKE '%word' → use FULLTEXT
  • col = 123 vs VARCHAR → type mismatch
  • NOT IN / != → rewrite with IN
  • OR on different cols → use UNION
  • LOWER(col) = 'x' → use collation
⚡ Pro Tips
  • CONCURRENTLY — no lock in Postgres
  • Partial indexes — subset of rows
  • INCLUDE — covering without key
  • Monitor unused indexes — drop them
  • Rebuild after bulk load — faster
  • ANALYZE / UPDATE STATS — keep fresh
